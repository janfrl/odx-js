title: Authentication

ODX supports various authentication methods for connecting to SAP and non-SAP backends. All credentials are managed server-side by the proxy — they are never exposed to the browser.

## Basic Authentication

Provide credentials directly in the service configuration. Ideal for development against NetWeaver systems.

```ts [nuxt.config.ts]
services: [
  {
    name: 'V2Service',
    url: 'https://my-sap-system.example.com/sap/opu/odata/...',
    auth: {
      username: 'USER',
      password: 'PASSWORD'
    }
  }
]
```

## Bearer Token

For systems using OAuth 2.0, you can pass a static bearer token.

```ts [nuxt.config.ts]
auth: {
  bearerToken: 'ey...'
}
```

## SAP BTP Principal Propagation

When deployed to SAP BTP, ODX automatically handles principal propagation if a `destination` is specified and the application is bound to the Destination and XSUAA services.

1. Specify the `destination` name in your config (must match the name in BTP Cockpit).
2. ODX uses `@sap-cloud-sdk/connectivity` to fetch the token and destination details at runtime.
3. The user's identity is propagated to the backend system.

```ts [nuxt.config.ts]
services: [
  {
    name: 'S4Products',
    destination: 'S4_BACKEND_DEST'
  }
]
```

For a full deployment walkthrough, see the [Deployment guide](/nuxt/deployment).

## CSRF Token Handling

SAP systems often require a CSRF token for state-changing requests (`POST`, `PATCH`, `DELETE`). ODX manages this automatically:

1. Before a state-changing request, ODX fetches a CSRF token via a `HEAD` or `GET` request to the service root.
2. The token is cached and reused for subsequent requests in the same session.
3. If a token is rejected (e.g., after session expiry), ODX automatically refreshes it.

No configuration is required — CSRF handling is active by default for all proxied services.
