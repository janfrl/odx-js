Die Integration von SAP OData in moderne TypeScript-Anwendungen ist oft mit Reibungsverlusten verbunden — magische Strings, manuelle Authentifizierungshandhabung und untypisierte Payloads. **ODX (OData Developer Experience)** verwandelt dies in einen typsicheren, modularen und visuell intuitiven Workflow.

## Was ist ODX?

ODX ist ein umfassendes Toolkit, das entwickelt wurde, um die Lücke zwischen Unternehmens-OData-Services und modernen Web-Frameworks wie Nuxt 4 zu schließen. Es übernimmt die schwere Arbeit der Authentifizierung, Typgenerierung und des Request-Proxyings, sodass Sie sich auf den Aufbau Ihrer Anwendung konzentrieren können.

## Systemarchitektur

ODX bietet einen sicheren, typsicheren Tunnel für den OData-Verkehr, der an die Anforderungen Ihres Projekts angepasst werden kann.

### Die Client-Schicht
Sie können OData auf zwei Arten konsumieren:
- **Mit Nuxt**: Verwenden Sie das `useOData` Composable für eine hochgradig reaktive Erfahrung mit vollständiger IDE-Autovervollständigung.
- **Direkt mit Core**: Verwenden Sie den Framework-agnostischen `$odata` Client oder gemeinsam genutzte Typen in jeder TypeScript-Umgebung (React, Svelte oder einfaches Node.js).

### Die Proxy-Schicht (Nitro)
Die meisten Unternehmens-OData-Anfragen werden über einen serverseitigen Proxy geleitet. Diese Schicht übernimmt die folgenden Aufgaben:
- **Authentifizierung**: Verwaltet nahtlos SAP BTP Destinationen, Basic- oder Bearer-Authentifizierung.
- **Sicherheitsrichtlinien**: Erzwingt Einschränkungen für Methoden, Pfade und Benutzerbereiche auf Serverebene.
- **CSRF-Schutz**: Übernimmt automatisch das SAP-spezifische Token-Pre-Fetching und die Rotation.

## Der ODX Explorer

Der Explorer ist eine voll funktionsfähige Anwendung zur Introspektion von OData. Er fungiert als integrierte **DevTools** für die Schema-Inspektion und das Traffic-Tracing während der Entwicklung und kann auch als eigenständiger **Service-Katalog** für Geschäftsanwender zum Durchsuchen und Testen verfügbarer APIs bereitgestellt werden.

## Der Request-Lebenszyklus

Ein typischer geproxter Request durchläuft die folgenden Phasen:

:::steps{level="3"}

### Composable-Aufruf
Ihre Anwendung ruft den OData-Client auf (z. B. `useOData().MyService.Products.list()`).

### Nuxt / Core Übersetzung
Der Aufruf wird in eine typisierte Fetch-Anfrage an entweder die interne Proxy-Route oder direkt an das Backend übersetzt.

### Proxy-Handler
Der ODX Proxy empfängt die Anfrage, wendet Sicherheitsrichtlinien an, verarbeitet CSRF-Token und leitet den Aufruf an SAP weiter.

### SAP Backend
Das Backend verarbeitet die OData-Anfrage und gibt die Antwort zurück.

### Antwort-Parsing
Die Antwort wird mit **Core SDK** Modellen typisiert und an Ihre Anwendung zurückgegeben.

:::

## Kernphilosophien

- **Typsicherheit zuerst**: Automatisierte Modellgenerierung aus Ihrem tatsächlichen SAP-Schema (EDMX).
- **Agnostisches Herz**: Die Core-Logik funktioniert in jeder TypeScript-Umgebung, nicht nur in Nuxt.
- **Unternehmenstauglich**: Erstklassige Unterstützung für SAP BTP, Principal Propagation und den Connectivity Service.
- **Offline-fähig**: Mocken Sie jedes OData-Backend mit lokalen JSON-Dateien für eine schnelle UI-Entwicklung.
