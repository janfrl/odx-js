`@bc8-odx/proxy` stellt die serverseitige Schicht für ODX bereit und übernimmt die Authentifizierung, das CSRF-Pre-Fetching und das Proxying von Anfragen an SAP-Backends.

## Features

- **BTP Destination Support**: Integriert mit `@sap-cloud-sdk/connectivity`, um Destinationen zur Laufzeit aufzulösen.
- **Sicherheitsrichtlinien**: Feingliedrige Kontrolle über erlaubte Methoden, Pfade und Header.
- **CSRF-Management**: Ruft automatisch SAP-CSRF-Token ab und rotiert diese.
- **Streaming-Unterstützung**: Leichtgewichtiges Proxying mit minimalem Speicherverbrauch.

## `createODataHandler(config)`

Erstellt einen Standard H3 Event-Handler, der OData-Proxy-Anfragen verarbeitet.

```ts
import { createODataHandler } from '@bc8-odx/proxy'

const handler = createODataHandler({
  services: [
    { name: 'MyService', url: '...', destination: 'MY_DESTINATION' }
  ]
})
```

## Nutzung als Nitro-Modul

```ts [nitro.config.ts]
export default defineNitroConfig({
  modules: ['@bc8-odx/proxy/nitro'],
  odata: {
    services: [
      { name: 'MyService', url: '...', destination: 'MY_DESTINATION' }
    ]
  }
})
```

## BTP Destination-Optionen

| Option | Typ | Beschreibung |
| --- | --- | --- |
| `destination` | `string` | Der Name der BTP-verwalteten Destination. |
| `proxyMode` | `'stream' \| 'buffer'` | Modus der Antwortverarbeitung. Standard: `stream`. |

## Interne Explorer-Endpunkte

`@bc8-odx/proxy` besitzt die interne `/__odx__` Laufzeit-API, die vom Explorer
genutzt wird. Diese Endpunkte sind keine oeffentlichen Produkt-APIs.

In Produktion leitet der AppRouter nur
`/__odx__/{config,logs,schema,generate,types,me}` an den Proxy weiter und XSUAA
ist erforderlich. Der Proxy validiert zusaetzlich den SAP Security Context.

Produktions-`/__odx__/config` gibt nur Top-Level `basePath`, `mode` und
`services` zurueck. Service-Eintraege sind auf `name`, `route`, `icon`,
`strategy`, `proxyMode`, `entities`, `isGenerated` und `version` begrenzt;
Backend-URLs, Destinations, Auth, Header, Rules, unbekannte Felder, Secrets,
Laufzeitpfade, Hooks, DevTools-Konfiguration, `forwardAuthHeader` und
`versions.node` werden weggelassen.

`/__odx__/logs` gibt in Produktion `[]` zurueck und lehnt Loeschen ab, sofern
SQL-Log-Speicherung nicht explizit konfiguriert ist. Mit SQL-Speicherung nutzt
der Proxy seinen db0-backed `OdxLogStore`-Adapter, um redigierte
Traffic-Historie aufzulisten und zu loeschen; Request- und
Response-Payload-Bodies werden standardmaessig weggelassen.
Produktions-Schema-Zugriff nutzt geparste gecachte Metadaten,
`/__odx__/generate` aktualisiert nur den Laufzeit-Metadatencache, und
`/__odx__/types` gibt `403` zurueck.

## Nitro-Hooks

| Hook | Beschreibung |
| --- | --- |
| `odx:proxy:request` | Wird vor jeder geproxten Anfrage ausgelöst (global). |
| `odx:proxy:request:[ServiceName]` | Wird vor Anfragen an einen bestimmten Service ausgelöst. |
| `odx:proxy:response` | Wird nach Erhalt der Backend-Antwort ausgelöst. |
