Der ODX Proxy ist für eine tiefe Integration in Cloud-Umgebungen konzipiert. Derzeit bietet er spezialisierte Unterstützung für die **SAP Business Technology Platform (BTP)**.

## SAP BTP (Business Technology Platform)

Beim Betrieb auf SAP BTP nutzt der Proxy Standard-Plattformservices für die Authentifizierung, Konnektivität und das Konfigurationsmanagement.

### Destination Service

Der Proxy kann Backend-URLs und technische Anmeldedaten automatisch über den BTP Destination Service auflösen.

**Voraussetzung**: Binden Sie eine Instanz des **Destination Service** und des **XSUAA Service** an Ihre Anwendung.

**Verwendung**: Verwenden Sie in Ihrer Service-Konfiguration den Namen der BTP Destination als Eigenschaft `destination`.

```ts
{
  services: [
    {
      name: 'S4ProductService',
      destination: 'S4_BACKEND_DEST' // Muss exakt mit dem Namen im BTP Cockpit übereinstimmen
    }
  ]
}
```

Der Proxy wird automatisch:

- Die Destination-Konfiguration abrufen und die URL auflösen.
- Die **Principal Propagation** handhaben, falls ein Benutzer-JWT vorhanden ist.
- Den **Connectivity Service** verwenden, falls das Ziel als `OnPremise` konfiguriert ist.

### Laufzeitkonfiguration (CUPS) {#runtime-configuration-cups}

Für Produktionsszenarien, in denen Sie Sicherheitsregeln oder Service-Einstellungen ohne ein erneutes Deployment aktualisieren müssen, verwenden Sie einen **User-Provided Service** (CUPS).

1. Erstellen Sie eine User-Provided Service-Instanz namens **`odx-config`**.
2. Geben Sie ein JSON-Objekt in den Credentials an:

```json
{
  "services": {
    "MyService": {
      "url": "https://new-backend-url.com",
      "rules": [
        { "type": "allowOnlyMethods", "value": ["GET"] },
        { "type": "requireScope", "value": "Admin" }
      ]
    }
  }
}
```

3. **Starten Sie die Anwendung neu**. Der Proxy erkennt diese Bindung automatisch und führt die Einstellungen zur Laufzeit mit Ihrer Konfiguration zusammen.

### Security Context (XSUAA)

Der Proxy initialisiert in der Produktionsumgebung automatisch einen Security Context für jede Anfrage. Dies ermöglicht die Verwendung von [Sicherheits-Helpern](/de/proxy/policies#security-helpers) wie `requireScope` oder `requireAttribute`.

- **JWT-Validierung**: Validiert automatisch den `Authorization`-Header gegen Ihre gebundene XSUAA-Instanz.
- **Attribute**: Greift auf benutzerdefinierte SAML-Attribute aus dem Identity-Token für eine erweiterte Policy-Erzwingung zu.

Der gleiche Produktions-Security-Context ist fuer Explorer-Laufzeit-APIs unter
`/__odx__` erforderlich. Entwicklungs-DevTools-Endpunkte koennen lokale
Fallback-Identitaet verwenden, aber deployte Explorer-Aufrufe an
`/__odx__/config`, `/__odx__/logs`, `/__odx__/schema`, `/__odx__/generate`,
`/__odx__/types` und `/__odx__/me` muessen authentifiziert sein, bevor der
Proxy Laufzeitdaten zurueckgibt.

### Persistente Explorer-Logs

Produktions-Traffic-Historie im Explorer ist opt-in. Binden Sie eine
verwaltete SQL-Datenbank an den Proxy, stellen Sie die Verbindungs-URL ueber
die Laufzeitumgebung bereit, und setzen Sie
`NUXT_ODATA_DEVTOOLS_LOG_STORE=sql`. Der Proxy speichert redigierte Eintraege
ueber seinen db0-backed `OdxLogStore`-Adapter; der Explorer ruft weiterhin nur
die Same-Origin-API `/__odx__/logs` auf. SQLite ist fuer lokale Entwicklung
oder explizite Single-Instance-Demos gedacht, nicht fuer Multi-Instance-BTP-
Deployments.
