Der ODX Proxy enthält eine leistungsstarke, **automatisch getracete Policy Engine**, mit der Sie Sicherheitsregeln erzwingen und Anfragen über eine saubere, deklarative API transformieren können. Jede Regel-Auswertung wird automatisch in der **Proxy-Telemetrie** des ODX Explorers aufgezeichnet.

Da der Proxy für Production-Workloads leistungsstarkes Streaming verwendet, konzentriert sich die Engine primär auf die **Request-Phase**. Dies ermöglicht es Ihnen, Anfragen zu validieren oder zu ändern, bevor sie an das SAP-Backend weitergeleitet werden.

## Regeln anwenden

Policies können auf drei Arten angewendet werden: über [BTP Runtime Overrides](/de/proxy/providers#runtime-configuration-cups), deklarative Konfiguration oder programmatische Hooks.

### Deklarative Konfiguration

Standard-Sicherheitsregeln können direkt in Ihrer `nuxt.config.ts` (oder der eigenständigen Proxy-Konfiguration) definiert werden.

```ts
{
  odata: {
    services: [
      {
        name: 'BusinessPartner',
        url: '/sap/opu/odata/sap/API_BUSINESS_PARTNER',
        rules: [
          { type: 'allowOnlyMethods', value: ['GET', 'POST'] },
          { type: 'requireScope', value: 'Display' },
          { type: 'denyPath', value: '/$metadata' }
        ]
      }
    ]
  }
}
```

### Programmatische Hooks (Nitro Plugins)

Für komplexe Logik, die Berechnungen zur Laufzeit erfordert, verwenden Sie das Utility `odataGuard` in Nitro Plugins.

```typescript
import { odataGuard } from '@me-tools/odx-proxy'

export default defineNitroPlugin((nitro) => {
  nitro.hooks.hook('odx:proxy:request:MyService', async (ctx) => {
    odataGuard(ctx)
      .allowOnlyMethods(['GET', 'POST'])
      .requireScope('Admin')
  })
})
```

Die Engine bietet Hooks für `odx:proxy:request` (global) und `odx:proxy:request:[ServiceName]` (service-spezifisch).

## Sicherheits-Helper

Diese Helper integrieren sich in den BTP Security Context (XSUAA), um unbefugte Anfragen abzulehnen.

:::field-group
  :::field{name="requireScope(scope: string)"}
  Überprüft, ob der aktuelle Benutzer über den angegebenen XSUAA-Scope verfügt. Falls dieser fehlt, wird die Anfrage mit `403 Forbidden` abgelehnt.
  :::

  :::field{name="requireAttribute(name: string, value: string)"}
  Stellt sicher, dass ein bestimmtes Benutzerattribut (z. B. `CostCenter` oder `CompanyID`) im Identity-Token mit dem erforderlichen Wert übereinstimmt.
  :::

  :::field{name="denyIfPathIncludes(path: string, reason?: string)"}
  Blockiert den Zugriff auf bestimmte Entitäts-Sets oder Unterpfade innerhalb des OData-Services.
  Beispiel: `.denyIfPathIncludes('/Users', 'Zugriff auf Benutzerdaten ist eingeschränkt')`
  :::

  :::field{name="denyIfHeader(name: string, value: string, reason?: string)"}
  Lehnt die Anfrage ab, wenn ein bestimmter HTTP-Header mit einem verbotenen Wert übereinstimmt.
  :::
:::

## Methoden-Kontrolle

Steuert, welche HTTP-Aktionen für einen Service zulässig sind.

:::field-group
  :::field{name="allowOnlyMethods(methods: string[])"}
  Beschränkt den Service auf eine explizite Liste von Methoden. Jede andere Methode führt zu einem `405 Method Not Allowed`.
  Beispiel: `.allowOnlyMethods(['GET'])` — macht einen Service effektiv schreibgeschützt.
  :::

  :::field{name="denyMethods(methods: string[])"}
  Blockiert gezielt bestimmte Methoden (z. B. `DELETE`), während alle anderen standardmäßig erlaubt sind.
  :::
:::

## Transformations-Helper

Diese Methoden ändern die Anfrage, bevor sie an das Zielsystem gesendet wird.

:::field-group
  :::field{name="injectHeader(name: string, value: string)"}
  Fügt der ausgehenden Anfrage einen benutzerdefinierten HTTP-Header hinzu. Wird häufig verwendet, um technische API-Keys sicher von der Serverseite aus zu übergeben.
  :::

  :::field{name="rewritePath(pattern: string | RegExp, replacement: string)"}
  Ändert dynamisch den Pfad der Request-URL.
  Beispiel: `.rewritePath(/^\/v2/, '/sap/opu/odata/sap')`
  :::
:::

## Benutzerdefinierte Logik

### `validate(name: string, callback: fn, reason?: string)`

Ermöglicht beliebige Validierungslogik. Wenn der Callback `false` zurückgibt, wird die Anfrage abgelehnt.

```typescript
odataGuard(ctx).validate('BusinessHours', () => {
  const hour = new Date().getHours()
  return hour >= 8 && hour <= 18
}, 'Zugriff nur während der Geschäftszeiten erlaubt')
```

### `info(message: string, details?: any)`

Fügt einen nicht-blockierenden Informationseintrag zum Proxy-Trace hinzu, der im Panel "Proxy Telemetry" des Explorers sichtbar ist.
