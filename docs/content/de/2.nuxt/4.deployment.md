Das Deployment einer ODX-Anwendung auf SAP BTP umfasst die Vorbereitung Ihrer `mta.yaml` und die Sicherstellung, dass die erforderlichen Plattformservices gebunden sind.

## 1. Projekt vorbereiten

Stellen Sie sicher, dass das Modul `@bc8-odx/nuxt` in Ihrem Projekt installiert ist und ein Production-Build lokal erfolgreich durchläuft:

```bash
pnpm run build
```

## 2. Die `mta.yaml` Struktur

Ihre `mta.yaml` sollte die folgenden Ressourcen deklarieren:

- **Destination Service**: Zur Auflösung von BTP-verwalteten OData-Backends.
- **XSUAA Service**: Für Authentifizierung und Principal Propagation.
- Optional: **Managed HTML5 Repository**, wenn ein eigenständiger AppRouter verwendet wird.

```yaml [mta.yaml]
modules:
  - name: my-odx-app
    type: nodejs
    path: .
    requires:
      - name: my-destination-service
      - name: my-xsuaa-service

resources:
  - name: my-destination-service
    type: org.cloudfoundry.managed-service
    parameters:
      service: destination
      service-plan: lite

  - name: my-xsuaa-service
    type: org.cloudfoundry.managed-service
    parameters:
      service: xsuaa
      service-plan: application
```

## 3. Build und Deployment

```bash
# MTA-Archiv bauen
mbt build

# Auf Cloud Foundry deployen
cf deploy mta_archives/my-odx-app_0.0.1.mtar
```

## 4. Produktions-Explorer-Laufzeit

In der lokalen Entwicklung laeuft der Explorer als Nuxt DevTools Tab. Im
vollstaendigen BTP Deployment laeuft er als eigenstaendige App hinter dem
AppRouter. AppRouter schuetzt die Explorer UI und `/__odx__` Laufzeit-APIs mit
XSUAA; danach validiert der Proxy den SAP Security Context, bevor er
Laufzeitdaten zurueckgibt.

Aktuelle Produktions-Policies:

- `/__odx__/client` und `/__odx__/client/*` liefern die Explorer UI aus.
- Nur `/__odx__/{config,logs,schema,generate,types,me}` wird an die
  Proxy-Laufzeit-API geleitet.
- `/__odx__/config` gibt nur Top-Level `basePath`, `mode` und bereinigte
  `services` zurueck.
- Bereinigte Service-Eintraege enthalten nur `name`, `route`, `icon`,
  `strategy`, `proxyMode`, `entities`, `isGenerated`, `version` und
  `metadata`. Das `metadata`-Objekt ist auf Laufzeit-Cache-Status begrenzt.
- Backend-URLs, Destinations, Auth, ausgehende Header, Rules, unbekannte
  Service-Felder, globale Secrets, Laufzeitpfade, Hooks, DevTools-Konfiguration,
  `forwardAuthHeader` und `versions.node` werden weggelassen.
- Produktions-Schema-/Config-Ansichten nutzen geparste gecachte Metadaten und
  expliziten Metadatenstatus. Roh-XML wird abgelehnt.
- `/__odx__/generate?service=<name>` aktualisiert in Produktion nur den
  Laufzeit-Metadatencache. Der Endpunkt fuehrt kein `odata2ts` aus und schreibt
  keine generierten TypeScript-SDK-Dateien.
- `/__odx__/types` gibt `403` zurueck; generierte Typartifacts sind
  Entwicklungs-, Build- oder CI-Ausgabe und werden von der deployten Laufzeit
  nicht ausgeliefert.
- Produktionslogs nutzen die proxy-eigene `OdxLogStore`-Grenze. Ohne
  SQL-Log-Speicherung gibt `GET` `[]` zurueck und `DELETE` gibt `403` zurueck.
  Mit `devtools.logStore.provider=sql` oder
  `NUXT_ODATA_DEVTOOLS_LOG_STORE=sql` speichert und liefert der Proxy
  redigierte Traffic-Historie ueber seinen db0-backed SQL-Adapter.
  Produktions-Payload-Bodies werden standardmaessig weggelassen.

Laufzeit-Metadaten-Refresh ist von TypeScript SDK-Generierung getrennt. Die
Produktionsaktion Refresh Metadata aktualisiert nur den Explorer-Cache fuer
Schema-/Config-Ansichten; Typaenderungen in der Anwendung benoetigen weiterhin
Build- oder CI-Generierung und ein neues Deployment.

## 5. Fehlerbehebung

Wenn Backend-Aufrufe nach dem Deployment fehlschlagen:

- **Destination-Name**: Überprüfen Sie, ob der `destination`-Name in `nuxt.config.ts` exakt mit dem im BTP Cockpit übereinstimmt.
- **Service-Bindings**: Führen Sie `cf env my-odx-app` aus, um zu bestätigen, dass die Services Destination und XSUAA korrekt gebunden sind.
- **Anwendungs-Logs**: Überprüfen Sie die Logs mit `cf logs my-odx-app --recent` auf Fehler auf Proxy-Ebene.
- **Explorer-Laufzeit-APIs**: Pruefen Sie, ob die AppRouter-Route einer der
  unterstuetzten Endpunkte `/__odx__/{config,logs,schema,generate,types,me}` ist
  und ob die Anfrage SAP Security Context hat.
