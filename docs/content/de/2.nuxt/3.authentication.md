ODX unterstützt verschiedene Authentifizierungsmethoden für die Verbindung zu SAP- und Nicht-SAP-Backends. Alle Anmeldedaten werden serverseitig vom Proxy verwaltet — sie werden niemals dem Browser gegenüber offengelegt.

## Authentifizierungsmethoden

:::accordion
  :::accordion-item{label="Basic-Authentifizierung" icon="i-heroicons-user"}
  Geben Sie Anmeldedaten direkt in der Service-Konfiguration an. Ideal für die Entwicklung gegen NetWeaver-Systeme.

  ```ts [nuxt.config.ts]
  services: [
    {
      name: 'V2Service',
      auth: {
        username: 'USER',
        password: 'PASSWORD'
      }
    }
  ]
  ```
  :::

  :::accordion-item{label="Bearer-Token" icon="i-heroicons-key"}
  Für Systeme, die OAuth 2.0 verwenden, können Sie einen statischen Bearer-Token übergeben.

  ```ts [nuxt.config.ts]
  auth: {
    bearerToken: 'ey...'
  }
  ```
  :::

  :::accordion-item{label="SAP BTP Principal Propagation" icon="i-simple-icons-sap"}
  Bei Bereitstellung auf SAP BTP übernimmt ODX die Principal Propagation automatisch über die Destination- und XSUAA-Services.

  ::steps{level="3"}
  ### Destination definieren
  Geben Sie den `destination`-Namen in Ihrer Konfiguration an (muss mit dem Namen im BTP Cockpit übereinstimmen).

  ```ts [nuxt.config.ts]
  services: [
    {
      name: 'S4Products',
      destination: 'S4_BACKEND_DEST'
    }
  ]
  ```

  ### Services binden
  Stellen Sie sicher, dass Ihre Anwendung an die Services **Destination** und **XSUAA** in BTP gebunden ist.

  ### Identitätsweitergabe
  ODX verwendet `@sap-cloud-sdk/connectivity`, um den Token abzurufen und die Identität des Benutzers zur Laufzeit weiterzugeben.
  ::
  :::
:::

## CSRF-Token-Handhabung

SAP-Systeme erfordern oft einen CSRF-Token für zustandsverändernde Anfragen (`POST`, `PATCH`, `DELETE`). ODX verwaltet dies automatisch:

:::steps{level="3"}
  ### Pre-fetch
  Vor einer zustandsverändernden Anfrage ruft ODX einen CSRF-Token über eine `HEAD`- oder `GET`-Anfrage ab.

  ### Cache
  Der Token wird zwischengespeichert und für nachfolgende Anfragen in derselben Sitzung wiederverwendet.

  ### Automatischer Refresh
  Wenn ein Token abgelehnt wird (z. B. nach Ablauf der Sitzung), aktualisiert ODX ihn automatisch und wiederholt die Anfrage.
:::

Es ist keine Konfiguration erforderlich — die CSRF-Handhabung ist standardmäßig für alle geproxten Services aktiv.
