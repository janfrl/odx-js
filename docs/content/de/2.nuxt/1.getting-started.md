Die Integration von ODX in Ihr Nuxt 4 Projekt ist unkompliziert. Folgen Sie diesen Schritten, um von Null zu Ihrer ersten typsicheren OData-Anfrage zu gelangen.

:::steps{level="2"}

## Paket installieren

Fügen Sie `@me-tools/odx-nuxt` mit Ihrem bevorzugten Paketmanager hinzu.

:::code-group
```bash [pnpm]
pnpm add @me-tools/odx-nuxt
```
```bash [npm]
npm install @me-tools/odx-nuxt
```
```bash [yarn]
yarn add @me-tools/odx-nuxt
```
:::

## Modul registrieren

Fügen Sie das Modul zum `modules`-Array in Ihrer `nuxt.config.ts` hinzu.

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  modules: [
    '@me-tools/odx-nuxt'
  ]
})
```

## Konfigurieren Sie Ihre Services

Fügen Sie einen `odata`-Block hinzu, um Ihre OData-Services zu deklarieren. Der `name` wird als Schlüssel in der `useOData()` Punkt-Notation-API verwendet.

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  modules: ['@me-tools/odx-nuxt'],

  odata: {
    // Optional: der Basispfad für die interne ODX Proxy-Route
    basePath: '/api/odx',

    services: [
      {
        name: 'V2Service',
        url: 'https://services.odata.org/V2/Northwind/Northwind.svc/',
        strategy: 'proxied'
      }
    ]
  }
})
```

### Service-Strategie

| Strategie | Beschreibung |
| --- | --- |
| `proxied` (Standard) | Anfragen werden über Nitro geleitet. Übernimmt automatisch die Authentifizierung (Basic, Bearer, BTP) und CSRF. Empfohlen für SAP. |
| `direct` | Anfragen werden direkt vom Browser an das OData-Backend gesendet. Geeignet für öffentliche APIs oder wenn CORS bereits behandelt wird. |

### Sicherheitsregeln

Feingliedrige Sicherheitsrichtlinien können pro Service angewendet werden, um HTTP-Methoden einzuschränken oder bestimmte Pfade zu blockieren.

```ts [nuxt.config.ts]
services: [
  {
    name: 'AdminService',
    url: '...',
    rules: [
      { type: 'allowOnlyMethods', value: ['GET'] },
      { type: 'denyPath', value: '/$metadata' }
    ]
  }
]
```

### Konfigurationspriorität

Einstellungen werden aus mehreren Quellen in der folgenden Reihenfolge zusammengeführt (die höchste gewinnt):

1. **Umgebungsvariablen**: `NUXT_ODATA_SERVICES_<NAME>_URL`
2. **SAP BTP User-Provided Services**: Laufzeit-Overrides über BTP-Bindings.
3. **Lokale Konfiguration**: Einstellungen in `nuxt.config.ts`

## TypeScript-Typen generieren

Führen Sie den `prepare`-Befehl aus, um Ihre Service-Metadaten abzurufen und TypeScript-Typen zu generieren.

```bash
npx nuxi prepare
```

Dies erstellt ein `.odx/` Verzeichnis in Ihrem Projekt-Root, das die generierten Modelle und EDMX-Metadaten enthält. Führen Sie diesen Befehl erneut aus, wann immer sich das Backend-Schema ändert.

## Ihre ersten Daten abrufen

Sie können nun das `useOData` Composable in jeder Vue-Komponente oder Seite verwenden.

```vue [pages/products.vue]
<script setup lang="ts">
const { data, pending, error } = await useOData().V2Service.Products.list({
  $select: ['ProductID', 'ProductName'],
  $top: 10
})
</script>

<template>
  <ul>
    <li v-for="p in data" :key="p.ProductID">
      {{ p.ProductName }}
    </li>
  </ul>
</template>
```

:::

Lesen Sie als Nächstes den [Anwendungsleitfaden](/de/nuxt/usage) für die vollständige Composable-API oder springen Sie zu [Authentifizierung](/de/nuxt/authentication), um eine Verbindung zu einem gesicherten SAP-Backend herzustellen.
