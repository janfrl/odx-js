Der ODX Explorer hat zwei Laufzeitformen:

- In der lokalen Nuxt Entwicklung ist er in die Nuxt DevTools eingebettet und
  auf Debugging-Ergonomie ausgelegt.
- In SAP BTP Produktionsdeployments ist er eine eigenstaendige App hinter dem
  AppRouter und nutzt authentifizierte `/__odx__` Laufzeit-APIs des Proxys.

## Features

### Schema Visualizer

Visualisieren Sie Entitätsbeziehungen, Eigenschaften und Navigationspfade Ihrer OData-Services. Unterstützt sowohl V2- als auch V4-EDMX-Schemata.

- Durchsuchen Sie alle Entity-Typen und Complex-Typen.
- Zeigen Sie Property-Namen, Typen, Nullability und Schlüsselfelder an.
- Erkunden Sie Navigation-Properties und Assoziationen.

### Traffic Monitor

Überwachen Sie Live-OData-Anfragen, die von Ihrer Anwendung ausgelöst werden.

- Sehen Sie die rohe Request-URL, Methode und Header.
- Inspizieren Sie die Request-Payload für `POST`/`PATCH`-Anfragen.
- Zeigen Sie den Response-Status, Header und Body an.
- Filtern Sie nach Service oder Entitäts-Set.

Entwicklungs-Traffic-Logs muessen Secrets redigieren und grosse Payloads
begrenzen, bevor sie angezeigt, gespeichert, exportiert oder kopiert werden.
Behandeln Sie ausgehende Header, Auth-/Session-/CSRF-Daten, Request-Bodies,
Response-Bodies und Proxy-Traces als sensibel.

### Data Browser

Durchsuchen und filtern Sie Entitätsdaten direkt in den DevTools.

- Führen Sie Ad-hoc-Abfragen mit OData-Systemabfrageoptionen aus.
- Laden Sie Entitätsdaten als JSON für das [Offline-Mocking](/de/explorer/setup#5-export-mock-data) herunter.

### Proxy Telemetry

Inspizieren Sie den Trace der Policy Engine für jede geproxte Anfrage. Jede Auswertung einer Policy-Regel wird mit ihrem Ergebnis (`allowed` / `denied`) und dem Grund aufgezeichnet, was das Debuggen der Sicherheitskonfiguration erleichtert.

## Konfiguration

| Option | Typ | Standard | Beschreibung |
| --- | --- | --- | --- |
| `devtools.enabled` | `boolean` | `true` | Zeigt den ODX-Tab in den Nuxt DevTools an. |
| `devtools.maxLogs` | `number` | `100` | Maximale Anzahl der zu behaltenden Traffic-Log-Einträge. |

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  odata: {
    devtools: {
      enabled: true,
      maxLogs: 200
    }
  }
})
```

## Explorer-Laufzeitvergleich

Die Nuxt DevTools Integration und die deployte eigenstaendige UI teilen sich
dieselbe Explorer-Anwendung. Ihre Laufzeit-APIs unterscheiden sich bewusst:
Entwicklung priorisiert lokales Debugging, Produktion nutzt Same-Origin-APIs
ueber AppRouter und Proxy mit einem reduzierten, authentifizierten Datensatz.

| Faehigkeit | Lokale Entwicklung | Produktionsdeployment |
| --- | --- | --- |
| UI-Auslieferung | Nuxt DevTools Tab unter `/__odx__/client/`. | Eigenstaendige Explorer App hinter AppRouter/XSUAA. |
| API-Origin | Lokaler Host-Proxy und DevTools-only Verhalten. | Same-Origin-`/__odx__` Laufzeit-APIs, die der AppRouter an den Proxy routet. |
| Auth und Identitaet | Lokale Fallback-Identitaet kann verwendet werden, wenn verfuegbar. | SAP Security Context ist erforderlich, bevor Laufzeitdaten zurueckgegeben werden; User Context wird bereinigt. |
| Config | Aufgeloeste Entwicklungskonfiguration zum Debuggen. | Bereinigte Allowlist mit Service-Identitaet, Route, Strategie, Proxy-Modus, Entity-/Generierungsstatus, OData-Version und Laufzeit-Metadatencache-Status. |
| Schema | Geparstes EDMX aus lokalen Dateien oder Cache; Roh-XML kann lokal angefordert werden. | Nur geparste gecachte Metadaten mit Status `available`, `stale` oder `missing`; Roh-XML wird abgelehnt. |
| Metadatenaktion | SDK-Regenerierung kann laufen, wenn der Nuxt Generator vorhanden ist. Hosts ohne Generator geben `501` zurueck. | Refresh Metadata aktualisiert nur den Laufzeit-Metadatencache. Es fuehrt kein `odata2ts` aus und schreibt keine generierten SDK-Dateien. |
| Generierte Typen | Generierte TypeScript-Modelldateien koennen ueber `/__odx__/types` gelesen werden. | Generierte Typartifacts sind deaktiviert; `/__odx__/types` gibt `403` zurueck. |
| Traffic-Logs | Standardmaessig memory-backed `OdxLogStore`. Filter und `DELETE` unterstuetzen lokale Retention-Workflows. Payload-Vorschauen werden begrenzt, wenn sie aktiviert sind. | Deaktiviert, sofern SQL-Speicherung nicht konfiguriert ist. Mit SQL liefert der Proxy redigierte db0-backed Historie ueber `OdxLogStore`; Request- und Response-Payload-Bodies werden standardmaessig weggelassen. |
| Mock-Daten | Data Browser kann JSON exportieren; lokale Mock-Fixture-Dateien werden direkt im Workspace verwaltet. | Keine serverseitige Mock-Daten-Clear-API ist exponiert, damit Produktions-Explorer keine Runtime- oder Kundendaten loeschen koennen. |

## `/__odx__` Endpunkt-Policy

Der AppRouter leitet `/__odx__/client` und `/__odx__/client/*` an die Explorer
UI weiter. Nur `/__odx__/{config,logs,schema,generate,types,me}` wird an die
Proxy-Laufzeit-API geleitet. Jeder Produktionsaufruf dieser API benoetigt SAP
Security Context.

Produktions-`/__odx__/config` gibt die Top-Level-Felder `basePath`, `mode` und
`services` zurueck. Service-Eintraege sind auf `name`, `route`, `icon`,
`strategy`, `proxyMode`, `entities`, `isGenerated`, `version` und `metadata`
begrenzt. Das `metadata`-Objekt ist auf Laufzeit-Cache-Status begrenzt:
`status`, `source`, `stale`, `staleReason`, `refreshedAt`, `timestamp`, `hash`,
`bytes` und optionales `message`.

Die Produktions-Config laesst Backend-URLs, Destinations, Auth, ausgehende
Header, Rules, unbekannte Service-Felder, globale Secrets, Laufzeitpfade,
Hooks, DevTools-Konfiguration, `forwardAuthHeader` und `versions.node` weg.

| Endpunkt | Aktuelles Produktionsverhalten |
| --- | --- |
| `/__odx__/config` | Authentifiziert. Bereinigte Allowlist mit Schema-/Entitaetsstatus aus dem Laufzeit-Metadatencache. |
| `/__odx__/logs` | Authentifiziert. Ohne SQL-Speicherung gibt `GET` `[]` zurueck und Loeschen gibt `403` zurueck; mit SQL-Speicherung listet und loescht der Endpunkt redigierte Traffic-Historie ueber `OdxLogStore`. |
| `/__odx__/schema?service=<name>` | Authentifiziert. Geparstes gecachtes Schema mit Metadatenstatus `available`, `stale` oder `missing`; Roh-XML wird abgelehnt. |
| `/__odx__/generate?service=<name>` | Authentifiziert. Aktualisiert nur den Laufzeit-Metadatencache und gibt `operation: "metadata-refresh"` mit Stale-, Timestamp-, Hash-, Byte-Count- und Source-Details zurueck. Fuehrt kein `odata2ts` aus und schreibt keine generierten SDK-Dateien. |
| `/__odx__/types?service=<name>` | Authentifiziert, aber mit `403` deaktiviert. |
| `/__odx__/me` | Authentifiziert. Bereinigter SAP User Context ohne rohe Tokendaten. |

Es gibt keinen `/__odx__/mockdata` Laufzeitendpunkt. Die Verwaltung von
Mock-Fixture-Dateien bleibt ein lokales Entwicklungs-Workspace-Thema.

## Laufzeit-Refresh und generierte Typen

Das aktuelle Produktionsverhalten kann bereinigte Config und geparste gecachte
Metadaten inspizieren. Die Aktion Refresh Metadata kann `$metadata` ueber den
produktionskompatiblen Service-Resolution-Pfad des Proxys abrufen und den
Laufzeit-Metadatencache fuer Explorer-Schema-/Config-Ansichten aktualisieren.
Wenn das Backend nicht erreichbar ist, aber ein Cache-Eintrag existiert,
koennen Antworten die Metadaten als stale markieren.

Refresh Metadata ist keine TypeScript SDK-Generierung. Produktion kann keine
SDK-Dateien in der deployten Nitro-Laufzeit neu generieren. Typaenderungen
benoetigen weiterhin Entwicklungs-, Build- oder CI-Generierung und ein neues
Deployment.

Produktions-Traffic-Historie ist nur verfuegbar, wenn SQL-Log-Speicherung
explizit konfiguriert ist. Der Proxy besitzt die `OdxLogStore`-Grenze,
Redaction, retention-freundliche Filter, Clear-Verhalten und den db0-backed
Adapter; der Explorer nutzt die API `/__odx__/logs` und spricht nie direkt mit
db0 oder einer Datenbank. Produktions-Request- und Response-Payload-Bodies
werden standardmaessig weggelassen.

## Eigenständiger Explorer (`@bc8-odx/explorer`)

Das Explorer-Paket wird auch fuer die deployte eigenstaendige UI verwendet. In
Produktion sollte es Same-Origin-`/__odx__`-APIs ueber den AppRouter aufrufen,
nicht unauthentifizierte Backend-Metadaten-URLs.
