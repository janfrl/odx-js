Der ODX Explorer hat zwei Laufzeitformen:

- In der lokalen Nuxt Entwicklung ist er in die Nuxt DevTools eingebettet und
  auf Debugging-Ergonomie ausgelegt.
- In SAP BTP Produktionsdeployments ist er eine eigenstaendige App hinter dem
  AppRouter und nutzt authentifizierte `/__odx__` Laufzeit-APIs des Proxys.

## Features

### Schema Visualizer

Visualisieren Sie Entitätsbeziehungen, Eigenschaften und Navigationspfade Ihrer OData-Services. Unterstützt sowohl V2- als auch V4-EDMX-Schemata.

- Durchsuchen Sie alle Entity-Typen und Complex-Typen.
- Zeigen Sie Property-Namen, Typen, Nullability und Schlüsselfelder an.
- Erkunden Sie Navigation-Properties und Assoziationen.

### Traffic Monitor

Überwachen Sie Live-OData-Anfragen, die von Ihrer Anwendung ausgelöst werden.

- Sehen Sie die rohe Request-URL, Methode und Header.
- Inspizieren Sie die Request-Payload für `POST`/`PATCH`-Anfragen.
- Zeigen Sie den Response-Status, Header und Body an.
- Filtern Sie nach Service oder Entitäts-Set.

Entwicklungs-Traffic-Logs muessen Secrets redigieren und grosse Payloads
begrenzen, bevor sie angezeigt, gespeichert, exportiert oder kopiert werden.
Behandeln Sie ausgehende Header, Auth-/Session-/CSRF-Daten, Request-Bodies,
Response-Bodies und Proxy-Traces als sensibel.

### Data Browser

Durchsuchen und filtern Sie Entitätsdaten direkt in den DevTools.

- Führen Sie Ad-hoc-Abfragen mit OData-Systemabfrageoptionen aus.
- Laden Sie Entitätsdaten als JSON für das [Offline-Mocking](/de/explorer/setup#5-export-mock-data) herunter.

### Proxy Telemetry

Inspizieren Sie den Trace der Policy Engine für jede geproxte Anfrage. Jede Auswertung einer Policy-Regel wird mit ihrem Ergebnis (`allowed` / `denied`) und dem Grund aufgezeichnet, was das Debuggen der Sicherheitskonfiguration erleichtert.

## Konfiguration

| Option | Typ | Standard | Beschreibung |
| --- | --- | --- | --- |
| `devtools.enabled` | `boolean` | `true` | Zeigt den ODX-Tab in den Nuxt DevTools an. |
| `devtools.maxLogs` | `number` | `100` | Maximale Anzahl der zu behaltenden Traffic-Log-Einträge. |

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  odata: {
    devtools: {
      enabled: true,
      maxLogs: 200
    }
  }
})
```

## Laufzeit-API-Modi

| Oberflaeche | Lokale Entwicklung | Produktionsdeployment |
| --- | --- | --- |
| UI-Auslieferung | Nuxt DevTools Tab unter `/__odx__/client/`. | Eigenstaendiger Explorer hinter AppRouter/XSUAA. |
| Auth | Entwicklungs-DevTools mit lokaler Fallback-Identitaet, wenn verfuegbar. | SAP Security Context ist fuer `/__odx__` Laufzeit-APIs erforderlich. |
| Config | Vollstaendige aufgeloeste Entwicklungskonfiguration zum Debuggen. | Nur bereinigte Allowlist. |
| Logs | In-Memory-Traffic-Logs; `DELETE` loescht lokale Logs. | Deaktiviert: `GET` gibt `[]` zurueck, `DELETE` gibt `403` zurueck. |
| Metadaten | Geparstes Schema und lokaler Roh-XML-Zugriff zum Debuggen. | Nur geparstes gecachtes Schema; Roh-XML wird abgelehnt. |
| SDK/Typen | Lokale Generierung und generierte Typdateien sind verfuegbar. | Generierungs- und Typartifact-Endpunkte geben `403` zurueck. |

## Produktions-Policy fuer `/__odx__`

Der AppRouter leitet `/__odx__/client` und `/__odx__/client/*` an die Explorer
UI weiter. Nur `/__odx__/{config,logs,schema,generate,types,me}` wird an die
Proxy-Laufzeit-API geleitet. Jeder Produktionsaufruf dieser API benoetigt SAP
Security Context.

Produktions-`/__odx__/config` gibt die Top-Level-Felder `basePath`, `mode` und
`services` zurueck. Service-Eintraege sind auf `name`, `route`, `icon`,
`strategy`, `proxyMode`, `entities`, `isGenerated` und `version` begrenzt.

Die Produktions-Config laesst Backend-URLs, Destinations, Auth, ausgehende
Header, Rules, unbekannte Service-Felder, globale Secrets, Laufzeitpfade,
Hooks, DevTools-Konfiguration, `forwardAuthHeader` und `versions.node` weg.

| Endpunkt | Aktuelles Produktionsverhalten |
| --- | --- |
| `/__odx__/config` | Authentifiziert. Nur bereinigte Allowlist. |
| `/__odx__/logs` | Authentifiziert. Gibt `[]` zurueck; Loeschen gibt `403` zurueck. |
| `/__odx__/schema?service=<name>` | Authentifiziert. Nur geparstes gecachtes Schema; Roh-XML wird abgelehnt. |
| `/__odx__/generate?service=<name>` | Authentifiziert, aber mit `403` deaktiviert. |
| `/__odx__/types?service=<name>` | Authentifiziert, aber mit `403` deaktiviert. |
| `/__odx__/me` | Authentifiziert. Bereinigter SAP User Context ohne rohe Tokendaten. |

## Aktuell und geplant

Das aktuelle Produktionsverhalten kann bereinigte Config und geparste gecachte
Metadaten inspizieren. Es kann keine TypeScript SDK-Dateien in Produktion neu
generieren.

Geplante Folgearbeiten trennen Laufzeit-Metadaten-Refresh von SDK-Generierung.
Ein zukuenftiger Produktions-Refresh kann den Laufzeit-Metadatencache fuer
Explorer-Schema-/Config-Ansichten aktualisieren, waehrend SDK-/Typgenerierung
ein Entwicklungs-, Build- oder CI-Workflow bleibt.

Produktions-Traffic-Historie ist ebenfalls geplant, aber noch nicht
implementiert. Die naechste Log-Arbeit fuehrt eine `OdxLogStore`-Grenze und
einen db0-Adapter mit Redaction, Retention, Payload-Limits und Clear-Verhalten
ein, bevor Produktionslogs aktiviert werden.

## Eigenständiger Explorer (`@bc8-odx/explorer`)

Das Explorer-Paket wird auch fuer die deployte eigenstaendige UI verwendet. In
Produktion sollte es Same-Origin-`/__odx__`-APIs ueber den AppRouter aufrufen,
nicht unauthentifizierte Backend-Metadaten-URLs.
