The ODX Proxy includes a powerful, **auto-traced Policy Engine** that allows you to enforce security rules and transform requests using a clean, declarative API. Every rule evaluation is automatically recorded in the ODX Explorer's **Proxy Telemetry**.

Because the proxy uses high-performance streaming for production workloads, the engine primarily focuses on the **Request Phase**. This allows you to validate or alter requests before they are forwarded to the SAP backend.

## Applying Rules

Policies can be applied in three ways: via [BTP Runtime Overrides](/explanation/proxy-providers#runtime-configuration-cups), declarative configuration, or programmatic hooks.

### Declarative Configuration
Standard security rules can be defined directly within your `nuxt.config.ts`.

```ts
{
  odata: {
    services: [
      {
        name: 'BusinessPartner',
        url: '/sap/opu/odata/sap/API_BUSINESS_PARTNER',
        rules: [
          { type: 'allowOnlyMethods', value: ['GET', 'POST'] },
          { type: 'requireScope', value: 'Display' },
          { type: 'denyPath', value: '/$metadata' }
        ]
      }
    ]
  }
}
```

### Programmatic Hooks (Nitro Plugins)
For complex logic that requires runtime calculations, use the `odataGuard` utility inside Nitro plugins.

```typescript
import { odataGuard } from '@bc8-odx/proxy'

export default defineNitroPlugin((nitro) => {
  nitro.hooks.hook('odx:proxy:request:MyService', async (ctx) => {
    odataGuard(ctx)
      .allowOnlyMethods(['GET', 'POST'])
      .requireScope('Admin')
  })
})
```

The engine provides hooks for `odx:proxy:request` (global) and `odx:proxy:request:[ServiceName]` (service-specific).

## Security Helpers

These helpers integrate with the BTP Security Context (XSUAA) to reject unauthorized requests.

### `requireScope(scope: string)`
Verifies that the current user has the specified XSUAA scope. If missing, the request is rejected with a `403 Forbidden`.

### `requireAttribute(name: string, value: string)`
Ensures a specific user attribute (e.g., `CostCenter` or `CompanyID`) in the identity token matches the required value.

### `denyIfPathIncludes(path: string, reason?: string)`
Blocks access to specific entity sets or sub-paths within the OData service.
* **Example**: `.denyIfPathIncludes('/Users', 'Access to user data is restricted')`

### `denyIfHeader(name: string, value: string, reason?: string)`
Rejects the request if a specific HTTP header matches a forbidden value.

## Method Control

Controls which HTTP actions are permitted on a service.

### `allowOnlyMethods(methods: string[])`
Restricts the service to an explicit list of methods. Any other method will result in an error.
* **Example**: `.allowOnlyMethods(['GET'])` effectively makes a service read-only.

### `denyMethods(methods: string[])`
Specifically blocks certain methods (e.g., `DELETE`) while allowing all others by default.

## Transformation Helpers

These methods modify the request before it is sent to the target system.

### `injectHeader(name: string, value: string)`
Adds a custom HTTP header to the outgoing request. Commonly used to pass technical API keys securely.

### `rewritePath(pattern: string | RegExp, replacement: string)`
Dynamically modifies the request URL path.
* **Example**: `.rewritePath(/^\/v2/, '/sap/opu/odata/sap')`

## Custom Logic

### `validate(name: string, callback: fn, reason?: string)`
Allows for arbitrary validation logic. If the callback returns `false`, the request is denied.

```typescript
odataGuard(ctx).validate('BusinessHours', () => {
  const hour = new Date().getHours()
  return hour >= 8 && hour <= 18
}, 'Access only allowed during business hours')
```

### `info(message: string, details?: any)`
Adds a non-blocking informational entry to the proxy trace.
