Integrating SAP OData into modern TypeScript applications is often filled with friction — magic strings, manual auth handling, and untyped payloads. **ODX (OData Developer Experience)** transforms this into a type-safe, modular, and visually intuitive workflow.

## What is ODX?

ODX is a comprehensive toolkit designed to bridge the gap between enterprise OData services and modern web frameworks like Nuxt 4. It handles the heavy lifting of authentication, type generation, and request proxying, allowing you to focus on building your application.

## System Architecture

ODX provides a secure, type-safe tunnel for OData traffic that can be adapted to your project's needs.

### The Client Layer
You can consume OData in two ways:
- **With Nuxt**: Use the `useOData` composable for a high-level, reactive experience with full IDE autocomplete.
- **Directly with Core**: Use the framework-agnostic `$odata` client or shared types in any TypeScript environment (React, Svelte, or plain Node.js).

### The Proxy Layer (Nitro)
Most enterprise OData requests are routed through a server-side proxy. This layer handles the heavy lifting of:
- **Authentication**: Seamlessly manages SAP BTP Destinations, Basic, or Bearer auth.
- **Security Policies**: Enforces method, path, and user scope restrictions at the server level.
- **CSRF Protection**: Automatically handles SAP-specific token pre-fetching and rotation.

## The ODX Explorer

The Explorer is a full-featured OData introspection application. It functions as integrated **DevTools** for schema inspection and traffic tracing during development, and can also be deployed as a standalone **Service Catalog** for business users to browse and test available APIs.

## The Request Lifecycle

A typical proxied request flows through the following stages:

:::steps{level="3"}

### Composable Call
Your application calls the OData client (e.g. `useOData().MyService.Products.list()`).

### Nuxt / Core Translation
The call is translated into a typed fetch request to either the internal proxy route or directly to the backend.

### Proxy Handler
The ODX Proxy receives the request, applies security policies, handles CSRF tokens, and forwards the call to SAP.

### SAP Backend
The backend processes the OData request and returns the response.

### Response Parsing
The response is typed using **Core SDK** models and returned to your application.

:::

## Core Philosophies

- **Type Safety First**: Automated model generation from your actual SAP schema (EDMX).
- **Agnostic Heart**: Core logic works in any TypeScript environment, not just Nuxt.
- **Enterprise Ready**: First-class support for SAP BTP, Principal Propagation, and Connectivity Service.
- **Offline Capable**: Mock any OData backend with local JSON files for rapid UI development.
