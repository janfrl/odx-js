Integrating SAP OData into modern TypeScript applications is often filled with friction — magic strings, manual auth handling, and untyped payloads. **ODX (OData Developer Experience)** transforms this into a type-safe, modular, and visually intuitive workflow.

## What is ODX?

ODX is a comprehensive toolkit designed to bridge the gap between enterprise OData services and modern web frameworks like Nuxt 4. It handles the heavy lifting of authentication, type generation, and request proxying, allowing you to focus on building your application.

## System Architecture

ODX sits between your frontend application and your SAP backend, providing a secure, type-safe tunnel for OData traffic.

### 1. The Client Layer (Nuxt 4)
Your application consumes OData via the `useOData` composable, leveraging types generated directly from your service's metadata. This ensures full IDE autocomplete and compile-time safety.

### 2. The Proxy Layer (Nitro)
All requests are routed through a server-side proxy. This layer is critical for enterprise security:
- **Authentication**: Seamlessly handles SAP BTP Destinations, Basic, or Bearer auth.
- **Security Policies**: Restrict methods, paths, and user scopes at the server level.
- **CSRF Protection**: Automatically manages SAP-specific token pre-fetching and rotation.

### 3. The Shared Core
The logic for OData query building, response parsing, and CSRF handling is encapsulated in the **Core SDK**, which remains framework-agnostic and can be used in any TypeScript environment.

## Project Structure

ODX is organized as a monorepo, ensuring tight integration and shared testing across all components:

```bash [Monorepo Layout]
packages/
├── core/      # Shared OData types, utilities, and lightweight client
├── explorer/  # DevTools UI and standalone service catalog app
├── nuxt/      # Nuxt 4 module for zero-config integration
├── proxy/     # Server-side Nitro proxy and policy engine
└── approuter/ # SAP BTP AppRouter configuration (optional)
```

## The Request Lifecycle

A typical proxied request flows through the following stages:

:::steps{level="3"}

### Composable Call
Your Vue component calls `useOData().MyService.Products.list()`.

### Nuxt Module
`@bc8-odx/nuxt` translates the dot-notation call into a typed fetch request to the internal proxy route (e.g. `/api/odx/MyService/Products`).

### Proxy Handler
`@bc8-odx/proxy` receives the request, applies security policies, fetches a CSRF token if needed, and forwards the call to the SAP backend.

### SAP Backend
The backend processes the OData request and returns the response.

### Core SDK
`@bc8-odx/core` provides the shared `ODataQuery` and response types used across all packages.

:::
