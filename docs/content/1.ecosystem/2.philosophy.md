## Why OData?

OData (Open Data Protocol) is an OASIS standard that defines a set of best practices for building and consuming RESTful APIs. It is the lingua franca of SAP's enterprise services — from S/4HANA and SuccessFactors to BTP Business Services — making it the most impactful protocol to support well in a modern web toolkit.

## OData V2 and V4 Support

ODX treats both protocol versions as first-class citizens because the SAP ecosystem spans both generations.

### OData V2

OData V2 (introduced around 2010) is still the dominant protocol in on-premise SAP systems (NetWeaver, ECC, older S/4HANA releases). Its characteristics:

- XML-based `$metadata` (EDMX)
- `application/atom+xml` or `application/json` response format
- CSRF token protection for state-changing requests
- Cookie-based session management common on NetWeaver

ODX handles all V2-specific quirks automatically: CSRF pre-fetching, `sap-client` header injection, and `X-Requested-With` headers.

### OData V4

OData V4 (the current standard) is used by newer SAP BTP services and S/4HANA Cloud. Its characteristics:

- Richer query capabilities (`$apply`, `$compute`, cross-joins)
- JSON-first response format
- Modern auth patterns (OAuth 2.0, PKCE)
- No CSRF tokens required for most endpoints

ODX's core client is protocol-aware and selects the correct serialization and header strategy based on the `version` property in your service configuration.

## Framework Agnosticism

The `@bc8-odx/core` package is intentionally free of any framework dependency. This means:

- You can use the `$odata` client in a plain Node.js script, a React app, or a SvelteKit server action.
- The Nuxt module and proxy are *consumers* of core — they add framework-specific ergonomics on top, but never dictate the underlying behavior.

This layered architecture ensures that upgrading Nuxt or switching frameworks does not require a rewrite of your OData integration logic.

## Security as a Default

ODX is designed for enterprise environments where security is non-negotiable:

- **Proxied-by-default**: Service credentials never leave the server. The browser only talks to your own Nitro proxy.
- **CSRF protection**: Automatically managed for all state-changing requests.
- **Declarative policy engine**: Restrict methods, paths, and scopes per service without writing custom middleware.
- **BTP Security Context**: XSUAA JWT validation is built in when running on SAP BTP.

## Offline-First Development

Mocking OData backends for UI development is notoriously painful. ODX provides a first-class offline mode:

1. Export entity data as JSON from the Explorer.
2. Place the JSON files in your `.odx/mock/` directory.
3. Start your app — ODX intercepts the requests and serves the mock data without any code changes.
