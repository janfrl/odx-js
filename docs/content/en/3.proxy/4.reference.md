`@bc8-odx/proxy` provides the server-side layer for ODX, handling authentication, CSRF pre-fetching, and proxying requests to SAP backends.

## Features

- **BTP Destination Support**: Integrated with `@sap-cloud-sdk/connectivity` to resolve destinations at runtime.
- **Security Policies**: Fine-grained control over allowed methods, paths, and headers.
- **CSRF Management**: Automatically fetches and rotates SAP CSRF tokens.
- **Streaming Support**: Lightweight proxying with minimal memory overhead.

## `createODataHandler(config)`

Creates a standard H3 event handler that processes OData proxy requests.

```ts
import { createODataHandler } from '@bc8-odx/proxy'

const handler = createODataHandler({
  services: [
    { name: 'MyService', url: '...', destination: 'MY_DESTINATION' }
  ]
})
```

## Nitro Module Usage

```ts [nitro.config.ts]
export default defineNitroConfig({
  modules: ['@bc8-odx/proxy/nitro'],
  odata: {
    services: [
      { name: 'MyService', url: '...', destination: 'MY_DESTINATION' }
    ]
  }
})
```

## BTP Destination Options

| Option | Type | Description |
| --- | --- | --- |
| `destination` | `string` | The name of the BTP-managed destination. |
| `proxyMode` | `'stream' \| 'buffer'` | Response handling mode. Default: `stream`. |

## Internal Explorer Endpoints

`@bc8-odx/proxy` owns the internal `/__odx__` runtime API used by Explorer.
These endpoints are not public product APIs.

In production, AppRouter routes only
`/__odx__/{config,logs,schema,generate,types,me}` to the proxy and XSUAA is
required. The proxy also validates SAP security context.

Production `/__odx__/config` returns only top-level `basePath`, `mode`, and
`services`. Service entries are limited to `name`, `route`, `icon`, `strategy`,
`proxyMode`, `entities`, `isGenerated`, and `version`; backend URLs,
destinations, auth, headers, rules, unknown fields, secrets, runtime paths,
hooks, DevTools config, `forwardAuthHeader`, and `versions.node` are omitted.

`/__odx__/logs` returns `[]` and rejects clearing in production unless SQL log
storage is explicitly configured. With SQL storage enabled, the proxy uses its
db0-backed `OdxLogStore` adapter to list and clear redacted traffic history;
request and response payload bodies are omitted by default. Production schema
access uses parsed cached metadata, `/__odx__/generate` refreshes runtime
metadata cache state only, and `/__odx__/types` returns `403`.

## Nitro Hooks

| Hook | Description |
| --- | --- |
| `odx:proxy:request` | Fired before every proxied request (global). |
| `odx:proxy:request:[ServiceName]` | Fired before requests to a specific service. |
| `odx:proxy:response` | Fired after receiving a buffered backend response. |
| `odx:proxy:response:[ServiceName]` | Fired after receiving a buffered backend response for a specific service. |

Buffered response hooks may be asynchronous and are awaited before the proxied
request resolves. Stream proxy response hooks are not currently defined.
