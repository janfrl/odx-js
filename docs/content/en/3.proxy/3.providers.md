The ODX Proxy is designed to integrate deeply with cloud environments. Currently, it provides specialized support for **SAP Business Technology Platform (BTP)**.

## SAP BTP (Business Technology Platform)

When running on SAP BTP, the proxy leverages standard platform services to handle authentication, connectivity, and configuration management.

### Destination Service

The proxy can automatically resolve backend URLs and technical credentials using the BTP Destination Service.

**Requirement**: Bind a **Destination Service** and an **XSUAA Service** instance to your application.

**Usage**: In your service configuration, use the name of the BTP Destination as the `destination` property.

```ts
{
  services: [
    {
      name: 'S4ProductService',
      destination: 'S4_BACKEND_DEST' // Must match the name in BTP Cockpit
    }
  ]
}
```

The proxy will automatically:

- Fetch the destination configuration and resolve the URL.
- Handle **Principal Propagation** if a user JWT is present.
- Use the **Connectivity Service** if the target is configured as `OnPremise`.

### Runtime Configuration (CUPS) {#runtime-configuration-cups}

For production scenarios where you need to update security rules or service settings without a redeployment, use a **User-Provided Service** (CUPS).

1. Create a User-Provided Service instance named **`odx-config`**.
2. Provide a JSON object in the credentials:

```json
{
  "services": {
    "MyService": {
      "url": "https://new-backend-url.com",
      "rules": [
        { "type": "allowOnlyMethods", "value": ["GET"] },
        { "type": "requireScope", "value": "Admin" }
      ]
    }
  }
}
```

3. **Restart** the application. The proxy automatically detects this binding and merges the settings into your configuration at runtime.

### Security Context (XSUAA)

The proxy automatically initializes a security context for every request when running in production. This enables the use of [Security Helpers](/proxy/policies#security-helpers) like `requireScope` or `requireAttribute`.

- **JWT Validation**: Automatically validates the `Authorization` header against your bound XSUAA instance.
- **Attributes**: Accesses custom SAML attributes from the identity token for advanced policy enforcement.

The same production security context is required for Explorer runtime APIs
under `/__odx__`. Development DevTools endpoints may use local fallback
identity, but deployed Explorer calls to `/__odx__/config`, `/__odx__/logs`,
`/__odx__/schema`, `/__odx__/generate`, `/__odx__/types`, and `/__odx__/me`
must be authenticated before the proxy returns runtime data.

### Persistent Explorer Logs

Production Explorer traffic history is opt-in. Bind a managed SQL database to
the proxy, expose its connection URL through the runtime environment, and set
`NUXT_ODATA_DEVTOOLS_LOG_STORE=sql`. The proxy stores redacted entries through
its db0-backed `OdxLogStore` adapter; Explorer continues to call only the
same-origin `/__odx__/logs` API. SQLite is intended for local development or
explicit single-instance demos, not multi-instance BTP deployments.
