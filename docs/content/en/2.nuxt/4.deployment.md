Deploying an ODX application to SAP BTP involves preparing your `mta.yaml` and ensuring the necessary platform services are bound.

## 1. Prepare Your Project

Ensure your project has the `@bc8-odx/nuxt` module installed and a production build passes locally:

```bash
pnpm run build
```

## 2. The `mta.yaml` Structure

Your `mta.yaml` should declare the following resources:

- **Destination Service**: To resolve BTP-managed OData backends.
- **XSUAA Service**: For authentication and principal propagation.
- Optionally: **Managed HTML5 Repository** if using a standalone approuter.

```yaml [mta.yaml]
modules:
  - name: my-odx-app
    type: nodejs
    path: .
    requires:
      - name: my-destination-service
      - name: my-xsuaa-service

resources:
  - name: my-destination-service
    type: org.cloudfoundry.managed-service
    parameters:
      service: destination
      service-plan: lite

  - name: my-xsuaa-service
    type: org.cloudfoundry.managed-service
    parameters:
      service: xsuaa
      service-plan: application
```

## 3. Build and Deploy

```bash
# Build the MTA archive
mbt build

# Deploy to Cloud Foundry
cf deploy mta_archives/my-odx-app_0.0.1.mtar
```

## 4. Production Explorer Runtime

In local development the Explorer runs as a Nuxt DevTools tab. In the full BTP
deployment it runs as a standalone app behind the AppRouter. AppRouter protects
the Explorer UI and `/__odx__` runtime APIs with XSUAA, then the proxy validates
SAP security context before returning runtime data.

Current production policies:

- `/__odx__/client` and `/__odx__/client/*` serve the Explorer UI.
- Only `/__odx__/{config,logs,schema,generate,types,me}` is routed to the
  proxy runtime API.
- `/__odx__/config` returns only top-level `basePath`, `mode`, and sanitized
  `services`.
- Sanitized service entries expose `name`, `route`, `icon`, `strategy`,
  `proxyMode`, `entities`, `isGenerated`, and `version` only.
- Backend URLs, destinations, auth, outbound headers, rules, unknown service
  fields, global secrets, runtime paths, hooks, DevTools config,
  `forwardAuthHeader`, and `versions.node` are omitted.
- Production schema/config views use cached parsed metadata and explicit
  metadata status. Raw XML is rejected.
- `/__odx__/generate?service=<name>` refreshes runtime metadata cache state in
  production. It does not run `odata2ts` and does not write generated
  TypeScript SDK files.
- `/__odx__/types` returns `403`; generated type artifacts are development,
  build, or CI output and are not exposed by the deployed runtime.
- Production logs use the proxy-owned `OdxLogStore` boundary. Without SQL log
  storage, `GET` returns `[]` and `DELETE` returns `403`. With
  `devtools.logStore.provider=sql` or `NUXT_ODATA_DEVTOOLS_LOG_STORE=sql`,
  the proxy stores and serves redacted traffic history through its db0-backed
  SQL adapter. Production payload bodies are omitted by default.

Runtime metadata refresh is separate from TypeScript SDK generation. The
production Refresh Metadata action updates Explorer schema/config cache state
only; application type changes still require build or CI generation and a new
deployment.

## 5. Troubleshooting

If backend calls fail after deployment:

- **Destination name**: Verify that the `destination` name in `nuxt.config.ts` exactly matches the one in BTP Cockpit.
- **Service bindings**: Run `cf env my-odx-app` to confirm the Destination and XSUAA services are bound correctly.
- **Application logs**: Check logs with `cf logs my-odx-app --recent` for proxy-level errors.
- **Explorer runtime APIs**: Verify that the AppRouter route is one of the
  supported `/__odx__/{config,logs,schema,generate,types,me}` endpoints and
  that the request has SAP security context.
