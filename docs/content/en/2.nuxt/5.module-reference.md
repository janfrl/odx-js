The `@bc8-odx/nuxt` module is configured via the `odata` block in `nuxt.config.ts`.

## Module Options

::api-ref
---
name: ModuleOptions
---
::

## `odata.basePath`

| Type | Default |
| --- | --- |
| `string` | `'/api/odx'` |

The Nitro route prefix under which the ODX proxy handler is registered. All proxied service requests are forwarded through this path.

## `odata.services`

An array of service definitions. Each entry describes one OData backend.

| Property | Type | Required | Description |
| --- | --- | --- | --- |
| `name` | `string` | Yes | Unique identifier. Preserved as the generated registry key and used in `useOData()`. Use `useOData('service-name')` or bracket access for names that are not valid TypeScript identifiers. Avoid path separator characters because service names are also used in generated output and metadata cache paths. |
| `url` | `string` | Yes* | Base URL of the OData service. |
| `destination` | `string` | No | BTP Destination name (overrides `url` on BTP). |
| `strategy` | `'proxied' \| 'direct'` | No | Default: `proxied`. |
| `version` | `'v2' \| 'v4'` | No | OData protocol version. Auto-detected from `$metadata` if omitted. |
| `auth` | `AuthOptions` | No | Static credentials for Basic or Bearer auth. |
| `rules` | `PolicyRule[]` | No | Declarative security policies. |

*Either `url` or `destination` is required.

## `odata.devtools`

Controls the ODX Explorer integration in Nuxt DevTools during local
development. These options do not enable production traffic logging or bypass
production `/__odx__` authentication and redaction policy.

| Property | Type | Default | Description |
| --- | --- | --- | --- |
| `enabled` | `boolean` | `true` | Enable the Explorer panel. |
| `maxLogs` | `number` | `100` | Maximum number of traffic entries to retain. |
| `logPayloads` | `boolean` | `true` | Store bounded request/response payload previews in development traffic logs. Production payload bodies are omitted by default. |
| `maxPayloadBytes` | `number` | `32768` | Maximum serialized bytes kept per logged payload before replacing it with a truncated preview marker. |
| `logStore.provider` | `'memory' \| 'sql'` | `'memory'` | Traffic log storage provider. SQL storage is implemented in the proxy through db0. |
| `logStore.sql.connector` | `'postgresql' \| 'sqlite'` | Inferred when possible | SQL connector for persistent traffic logs. Use PostgreSQL or another managed network SQL service for BTP production. |
| `logStore.sql.url` | `string` | none | Database URL for network SQL providers. Keep credentials in environment variables or service bindings. |
| `logStore.sql.path` | `string` | none | SQLite database path for local development or explicit single-instance demos only. |

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  odata: {
    devtools: {
      enabled: true,
      maxLogs: 200
    }
  }
})
```

Development log entries are diagnostic data. Redact secrets in outbound
headers, auth/session/CSRF values, and request or response bodies, and bound
large payloads before exporting or copying them. Production Explorer traffic
history remains disabled unless SQL storage is explicitly configured through
`devtools.logStore.provider=sql` or `NUXT_ODATA_DEVTOOLS_LOG_STORE=sql`.
Persistent production entries are still redacted and omit request/response
payload bodies by default.

## Environment Variable Overrides

Any service property can be overridden at runtime via environment variables:

```bash
NUXT_ODATA_SERVICES_V2SERVICE_URL=https://new-backend.example.com
NUXT_ODATA_SERVICES_V2SERVICE_AUTH_USERNAME=admin
```

The pattern is `NUXT_ODATA_SERVICES_<NAME>_<PROPERTY>` where `<NAME>` is the uppercase service name.
