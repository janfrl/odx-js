The ODX Explorer has two runtime shapes:

- In local Nuxt development, it is embedded in Nuxt DevTools and favors
  debugging ergonomics.
- In SAP BTP production deployments, it is a standalone app served behind the
  AppRouter and uses authenticated `/__odx__` runtime APIs from the proxy.

## Features

### Schema Visualizer

Visualize the entity relationships, properties, and navigation paths of your OData services. Supports both V2 and V4 EDMX schemas.

- Browse all entity types and complex types.
- View property names, types, nullability, and key fields.
- Explore navigation properties and associations.

### Traffic Monitor

Monitor live OData requests triggered by your application.

- See the raw request URL, method, and headers.
- Inspect the request payload for `POST`/`PATCH` requests.
- View the response status, headers, and body.
- Filter by service or entity set.

Development traffic logs must redact secrets and limit large payloads before
they are displayed, stored, exported, or copied. Treat outbound headers,
auth/session/CSRF data, request bodies, response bodies, and proxy traces as
sensitive.

### Data Browser

Browse and filter entity data directly within the DevTools.

- Run ad-hoc queries with OData system query options.
- Download entity data as JSON for [offline mocking](/explorer/setup#5-export-mock-data).

### Proxy Telemetry

Inspect the Policy Engine trace for each proxied request. Each policy rule evaluation is recorded with its outcome (`allowed` / `denied`) and reason, making it easy to debug security configuration.

## Configuration

| Option | Type | Default | Description |
| --- | --- | --- | --- |
| `devtools.enabled` | `boolean` | `true` | Show the ODX tab in Nuxt DevTools. |
| `devtools.maxLogs` | `number` | `100` | Maximum number of traffic log entries to retain. |

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  odata: {
    devtools: {
      enabled: true,
      maxLogs: 200
    }
  }
})
```

## Runtime API Modes

| Surface | Local development | Production deployment |
| --- | --- | --- |
| UI delivery | Nuxt DevTools tab served from `/__odx__/client/`. | Standalone Explorer served behind AppRouter/XSUAA. |
| Auth | Development DevTools behavior with local fallback identity where available. | SAP security context is required for `/__odx__` runtime APIs. |
| Config | Full resolved development config for debugging. | Sanitized allowlist only. |
| Logs | In-memory traffic logs; `DELETE` clears local logs. | Disabled: `GET` returns `[]`, `DELETE` returns `403`. |
| Metadata | Parsed schema and local raw XML access for debugging. | Parsed cached schema only; raw XML is rejected. |
| SDK/types | Local generation and generated type files are available. | Generation and type artifact endpoints return `403`. |

## Production `/__odx__` Policy

The AppRouter routes `/__odx__/client` and `/__odx__/client/*` to the Explorer
UI. It routes only `/__odx__/{config,logs,schema,generate,types,me}` to the
proxy runtime API. Every production runtime API call requires SAP security
context.

Production `/__odx__/config` returns top-level `basePath`, `mode`, and
`services`. Service entries are limited to `name`, `route`, `icon`, `strategy`,
`proxyMode`, `entities`, `isGenerated`, and `version`.

The production config response omits backend URLs, destinations, auth, outbound
headers, rules, unknown service fields, global secrets, runtime paths, hooks,
DevTools config, `forwardAuthHeader`, and `versions.node`.

| Endpoint | Current production behavior |
| --- | --- |
| `/__odx__/config` | Authenticated. Sanitized allowlist only. |
| `/__odx__/logs` | Authenticated. Returns `[]`; clearing returns `403`. |
| `/__odx__/schema?service=<name>` | Authenticated. Parsed cached schema only; raw XML rejected. |
| `/__odx__/generate?service=<name>` | Authenticated but disabled with `403`. |
| `/__odx__/types?service=<name>` | Authenticated but disabled with `403`. |
| `/__odx__/me` | Authenticated. Sanitized SAP user context without raw token data. |

## Current Versus Planned

Current production Explorer behavior can inspect sanitized config and cached
parsed metadata. It cannot regenerate TypeScript SDK files in production.

Planned follow-up work separates runtime metadata refresh from SDK generation.
A future production refresh may update runtime metadata cache state for
Explorer schema/config views, while SDK/type generation remains a development,
build, or CI workflow.

Production traffic history is also planned, but not implemented yet. The next
log work introduces an `OdxLogStore` boundary and a db0-backed adapter with
redaction, retention, payload limits, and clear behavior before production logs
are enabled.

## Standalone Explorer (`@bc8-odx/explorer`)

The Explorer package is also used for the deployed standalone UI. In production
it should call same-origin `/__odx__` APIs through the AppRouter, not
unauthenticated backend metadata URLs.
